# Security Baseline

## Principles

1. **Defence in depth** — multiple layers of security controls
2. **Least privilege** — services only access what they need
3. **Secrets outside source** — no credentials in git-tracked files
4. **Central identity** — Keycloak is the single source of truth for authentication

## Network Security

### Docker Compose

- **Only the Caddy gateway is exposed** on `0.0.0.0:80`
- All other services use `expose` (Docker-internal only)
- Database ports (`5432`, `3306`) are never mapped to the host
- Redis ports (`6379`) are never mapped to the host
- Inter-service communication happens on the `mymusic_coach_network` bridge

### Production Considerations

- Enable TLS termination at the gateway (Caddy auto-HTTPS in production)
- Use `sslRequired: all` in Keycloak realm settings
- Configure network policies in Kubernetes to restrict pod-to-pod traffic

## Secrets Management

### Local Development

- All secrets are in `.env` (git-ignored)
- `.env.example` contains placeholder values — never real credentials
- `pretix.cfg` is generated at runtime from env vars (not committed)
- LibreBooking `config.php` is generated at runtime from env vars
- Moodle config is generated by the Bitnami image from env vars

### What Must Be Changed Before Production

`.env.example` ships with **pre-generated random passwords** suitable for local development. Each secret has a comment `# ↑ Change this ... before deploying to production`. Before going live:

| Secret | Location | Notes |
|--------|----------|-------|
| `JWT_SECRET` | `.env` | Regenerate — min 256-bit random hex |
| `NEXTAUTH_SECRET` | `.env` | Regenerate — min 256-bit random hex |
| `POSTGRES_PASSWORD` | `.env` | Regenerate — strong, unique |
| `REDIS_PASSWORD` | `.env` | Regenerate — strong, unique |
| `KEYCLOAK_ADMIN_PASSWORD` | `.env` | Regenerate immediately |
| `MOODLE_ADMIN_PASSWORD` | `.env` | Regenerate immediately |
| `PRETIX_API_TOKEN` | `.env` | Generate in pretix admin panel |
| `PRETIX_WEBHOOK_SECRET` | `.env` | Regenerate — shared secret with pretix |
| `STRIPE_SECRET_KEY` | `.env` | Use your production Stripe key |
| All `*_OIDC_CLIENT_SECRET` | `.env` | Regenerate in Keycloak |

## Authentication & Authorisation

### Identity Flow

```
User → Browser → Caddy → Next.js → NextAuth.js → Keycloak OIDC
                                                      ↓
                                            id_token + access_token
                                                      ↓
                                      Apollo Client → GraphQL API
                                         (Authorization: ******
                                                      ↓
                                        authMiddleware → verifyToken
                                                      ↓
                                          context.user (id, role)
```

### Keycloak Clients

| Client | Purpose | Protocol | Flow |
|--------|---------|----------|------|
| `mymusic-coach-web` | Next.js frontend | OIDC | Authorization Code + PKCE |
| `mymusic-coach-api` | API service account | OIDC | Client Credentials |
| `moodle-oidc` | Moodle SSO | OIDC | Authorization Code + PKCE |
| `pretix-oidc` | pretix customer SSO | OIDC | Authorization Code + PKCE |
| `librebooking-saml` | LibreBooking SSO | OIDC | Authorization Code |

### JWT Configuration

- Access tokens: 15-minute expiry
- Refresh tokens: 30-day expiry
- Keycloak realm tokens: 15-minute access token lifespan

### Role-Based Access Control

| Role | Capabilities |
|------|-------------|
| `GUEST` | Browse free courses, search teachers, view events |
| `STUDENT` | Full access to paid content, bookings, social feed |
| `TEACHER` | Create courses, accept bookings, publish events |
| `ADMIN` | Platform management, teacher verification |

## API Security

- **Helmet** — HTTP security headers (CSP, X-Frame-Options, etc.)
- **CORS** — restricted to configured origins only
- **Body size limit** — 50 MB max for uploads
- **Auth middleware** — permissive (allows unauthenticated for public queries); resolvers enforce auth via `requireAuth()` / `requireRole()`
- **Input validation** — Prisma parameterised queries prevent SQL injection

## Dependency Security

- Pin Docker image versions (no `:latest` tags)
- Pin npm dependency versions
- Run `npm audit` regularly
- Use Dependabot or Renovate for automated updates

## Data Protection

- Passwords hashed with bcrypt (12 rounds)
- External identity mappings stored with referential integrity
- Personal data accessible only to the owning user and admins
- Keycloak handles email verification
